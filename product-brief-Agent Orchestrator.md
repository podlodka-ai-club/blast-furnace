# Product Brief: Agent Orchestrator (Hacker Sprint #1)

## Executive Summary

Мы делаем оркестратор агентной разработки за две недели Hacker Sprint #1. Он берёт задачу из трекера, прогоняет её через пайплайн `Intake → Assess → Plan → Develop → Quality gate → Review → Draft PR → Move to 'In review'`. Состояние процесса живёт в файлах-артефактах в репозитории, а не в памяти процесса. При старте каждого следующего этапа предыдущий артефакт детерминированно проверяется — без этой проверки переход не происходит.

Это не универсальный продукт для рынка. Это рабочая система, которой команда сможет реально пользоваться на своих задачах после спринта, и которая заодно показывает наш подход к построению agent-пайплайнов. Основная ценность — в нашем понимании как это работает, устройстве пайплайна, наборе артефактов, промптах и протоколе handoff-а между этапами.

## Архитектурный подход

Pipeline:
```
                           ┌──────────┐─────────────┐
                           ↓          ↑             ↑
Intake → Assess → Plan → Develop + Quality gate → Review → Draft PR → Move to 'In review'
  ↑         ↓      ↑                                          ↓
  └ clarify ┘      └──────────────── rework ──────────────────┘
```

Ключевые решения:

- **Handoff через файлы-артефакты** в `.orchestrator/runs/<id>/`. Между этапами никакого in-memory state.
- **YOLO-sandbox mode**: оркестратор не переспрашивает пользователя посреди работы — это YOLO-часть. Но агент **не имеет прямого доступа к репозиторию**. Всё, что касается git и репо (commit, push, работа с ветками, создание PR), выполняет исключительно детерминированный код оркестратора. Агент получает рабочую директорию как изолированную среду: он читает и пишет файлы, но git-операции ему недоступны физически — это ограничение на уровне sandbox-а (доступные инструменты / subprocess-изоляция), а не на уровне промпта. Видимые действия — через GitHub (комментарии, лейблы, статусы) и вывод в терминал.
- **Прогресс пишется в issue в трекере.** План, полученный на этапе Plan, записывается отдельным комментарием к issue. Старт каждой попытки rework тоже оформляется отдельным комментарием. Это даёт видимый снаружи ход работы и упрощает разбор прошедших прогонов.
- **Один локальный executor в MVP — Codex CLI / Claude Code CLI.**
- **Rework-twice**: допускаются две попытки переделать работу для каждого AI-этапа пайплайна.
- **Два отдельных набора ревьюеров.** Это два разных режима проверки с разным назначением:
  - **Judge Committee** работает на этапе код-ревью перед PR. В нём несколько ролей (Correctness / Style / Scope), каждый ревьюер — в своём контексте, со своими промптами.
  - **Attacker Squad** включается только тогда, когда человек отклонил PR и нужен rework. В нём другие роли (диагност — ищет причину отказа; prescriber — формулирует, что конкретно исправить в плане). Скорее всего это будет один вызов LLM с двумя шагами рассуждения.

## Чем отличаемся

1. **Два набора ревьюеров.** Один — код-ревью перед PR. Второй — включается только на rework, если человек отклонил результат. У них разные промпты, роли и входные данные.
2. **Отказ на входе, когда задача недоспецифицирована.** Оркестратор не пытается определить когда описания задачи недостаточно для успешного выполнения: он оставляет в issue комментарий с конкретными вопросами и останавливается. Явный отказ лучше, чем фальшивый PR.
3. **Запуск на подписках, не на API.** Оркестратор работает через Claude Code CLI / Codex CLI, которые используют пользовательскую подписку, а не API с оплатой за токены. Это принципиально снижает стоимость прогонов.
4. **Диагностика причины отказа перед rework.** (Пересекается с 1) Когда человек отклоняет PR, перед переделкой запускается отдельный этап анализа: система выясняет, что именно пошло не так, и формулирует конкретные правки для планировщика. Rework начинается не вслепую.
5. **Негативная память в планировщике.** Зафиксированные ошибки прошлых прогонов («так больше не делаем») подаются на вход планировщику. Система не повторяет одни и те же ошибки от прогона к прогону.
6. **Self-improvement loop.** По итогам прогонов оркестратор анализирует, где систематически ошибается, и запускает прогон по улучшению собственного кода. Система способна улучшать саму себя.
7. **LLM Substitution.** Если две попытки rework с одной моделью не дали результата — следующая попытка запускается на другой LLM. Это может повысить вероятность выполнения задачи.

## Инженерные принципы

### Детерминированный код вместо LLM

Всё, что можно надёжно сделать обычным кодом, не остаётся на усмотрение LLM. Механические операции дешевле, предсказуемее и проще отлаживаются. LLM остаётся там, где действительно нужно рассуждение.

Детерминированным кодом делаем как минимум:

- commit, push, создание PR;
- quality gate (линтеры, тесты);
- проверка JSON-схем артефактов;
- работа с ветками;
- transitions в трекере;
- сборка описания commit и PR по шаблону;
- правила блокировки (превышено количество попыток или время работы этапа пайплайна).

### Quality Gate

Проверки — насколько возможно таргетированные:
- targeted lint по изменённым файлам;
- используем полный набор тестов, не усложняем систему определением и прогоном только нужной части тестов;
- таймауты: `LINT_TIMEOUT=120s`, `TEST_TIMEOUT=180s`.

Это ориентир, всё настроим на первых прогонах.

### Установка зависимостей

Для упрощения сначала можно отдать LLM. Затем стоит подумать в сторону subprocess + кеш по hash lock-файла (`package-lock.json`, `poetry.lock`, `requirements.txt`). Повторная установка — только если действительно изменился lock.

### Формат обмена между этапами

- Базовый формат — JSON.
- На этапах, где LLM что-то нормализует (например, превращает произвольный issue в Plan), LLM отвечает структурированным JSON-ом.
- У каждого такого JSON есть обязательный набор полей и схема.
- Корректность артефакта проверяется детерминированно — по схеме.

### Validation плана

Валидация плана после того, как LLM его сделал — не свободная оценка, а фиксированный набор измерений:

- `task_completeness` — полнота шагов относительно задачи;
- `dependency_correctness` — корректный порядок;
- `key_links` — ссылки на ключевые файлы и сущности;
- `scope_sanity` — не вышли ли за рамки задачи;
- `must_haves` — обязательные пункты на месте;
- `regression_safety` — что может сломаться.

### Тесты до кода

По возможности тесты появляются раньше кода:

- Реализацию стоит разбивать минимум на два подшага: сначала написать или обновить тесты, потом писать код.
- Прохождение тестов проверяется детерминированно.

Это принцип, а не догма. Если в MVP какая-то задача становится от этого слишком дорогой, допускаем исключение.

### Observability

Все артефакты, запросы и ответы сохраняются:

- Артефакты складываем в `.orchestrator/runs/<id>/raw/`
- JSONL c логами в `.orchestrator/runs/<id>/`

### Доступ к репозиторию

Для доступа к GitHub (чтение issue, создание PR, комментарии, транзиции статусов) оркестратор читает токен из переменной окружения — например, `GITHUB_TOKEN`. Для MVP этого достаточно: токен хранится локально, не коммитится.

### Intake задач из трекера

Задачи берутся из трекера автоматически по явному правилу. Возможные варианты:

- по лейблу или тегу (например, `agent-ready`);
- по статусу (например, `ToDo`);
- по другой явной разметке.

Точное правило ещё выберем.

## Целевой репозиторий

Пайплайн работает с одним конкретным репозиторием. Скорее всего это форк существующего репо, в который мы докладываем нужную обвязку и документацию. Репозиторий должен быть специально подготовлен для AI-assisted работы:

- на первом этапе поддержим только GitHub;
- issue ведутся через GitHub Issues;
- в репозитории есть понятная структура, `AGENTS.md` / `CLAUDE.md` или эквивалент;
- тесты и линтеры запускаются одним предсказуемым способом.

Выбор конкретного репозитория — открытый вопрос (см. ниже).

## Рабочая директория: worktree vs clone

Нужно определиться, как пайплайн получает чистую копию репо для каждого прогона:

- `git worktree` — легче и быстрее, но файлы, которые лежат локально и не в git (`.env`, ключи, локальные конфиги), в worktree не попадают;
- clone в отдельную директорию (через `cp`, а не `git clone`) — тяжелее, зато позволяет скопировать нужные файлы локального окружения.

## Subprocess per run

Отдельный инженерный вопрос: запускать ли каждый прогон в своём subprocess-е. Плюсы — изоляция окружения, предсказуемая очистка, возможность жёстких таймаутов. Похожий подход есть в `mini-swe-agent` — стоит посмотреть, как это у них сделано, и взять, если оправдано.

Не утверждаем как решение, это вопрос для проверки.

## Критерии успеха

Практические, проверяемые признаки того, что получилось:

- **Пайплайн работает.** Вероятность прохождения до конца и принятия PR > 50%.
- **Понятная документация.** По документации можно быстро понять, как система устроена и как запустить оркестратор для своего проекта.
- **Реальное использование.** После спринта команда действительно использует оркестратор на своих задачах.

## Ориентиры из референсных репозиториев

Не копируем полностью, но берём конкретные идеи.

- **OpenAI Symphony** ([SPEC.md](https://github.com/openai/symphony/blob/main/SPEC.md)) — устройство репозитория, набор сущностей оркестрации, общая структура описания orchestration-объектов. Хорошая точка сверки, когда будем формализовать свои артефакты.
- **OpenHands Resolver** ([README](https://github.com/OpenHands/OpenHands/blob/main/openhands/resolver/README.md)) — процесс создания Access Token и работа с внешним репозиторием. Как минимум — переносим их инструкцию к себе в документацию.
- **mini-swe-agent** ([repo](https://github.com/SWE-agent/mini-swe-agent)) — смотрим, как у них устроен запуск каждого прогона в отдельном subprocess-е, и решаем, брать ли такой подход.
- **Archon** ([repo](https://github.com/coleam00/Archon/)) — как работать с локальной авторизацией, локальными подписками и локальным доступом к агентным инструментам. Хотим уметь так же; если не получится — fallback на token-based схему.
- **yolo-runner** ([repo](https://github.com/egv/yolo-runner/)) — разделение ответственности: отдельно модуль для работы с трекером задач, отдельно модуль, который запускает и координирует агентов. Забираем это разделение.

## Scope

### P0 — База MVP (обязательно)

- **T1** Пайплайн `Intake → Assess → Plan → Develop → Quality gate → Review → Draft PR → Move to 'In review'` с handoff между этапами через файл(ы)-артефакт(ы).
- **T2** Intake из GitHub Issues по явному правилу → рабочая директория → executor → quality gate → draft PR.
- **T3** Rework-twice на AI-этапах.
- **T4** YOLO-sandbox mode: агент работает без переспросов, без прямого доступа к репозиторию (ограничение на уровне sandbox-а, не промпта); видимые действия через GitHub (комментарии, лейблы, статусы); план и старт rework пишутся комментариями к issue.
- **T5** Один локальный executor — Codex CLI или Claude Code CLI. Если возможно — реализовать локальную авторизацию в духе Archon (без токена из env); если это окажется сложным, fallback на token-based схему.
- **T6** Детерминированные commit / push / PR, quality gate (lint + tests с таймаутами), проверка JSON-схем артефактов, сохранение сырых артефактов и JSONL-логов.
- **T7** Доступ к GitHub через токен из переменной окружения.

### P1

- **T8** Judge Committee: три роли (Correctness / Style / Scope), раздельные контексты на этапе Review.
- **T9** Валидация Plan по фиксированным измерениям.
- **T10** Отказ на входе при недоспецифицированном issue (недетерминированный completeness gate + авто-комментарий в issue).
- **T11** Тесты до кода на этапе Develop (разбиение на два подшага — тесты, потом код).
- **T12** Телеметрия прогона — сбор событий и LLM-вызовов в единый JSONL-лог в `.orchestrator/runs/<id>/`.

### P2

- **T13** Attacker Squad на rework после отклонения PR человеком.
- **T14** Post-run review — артефакт с разбором прошедшего прогона. Дополнительно: на этапе Assess оценивается вероятность успешного выполнения задачи и кладётся в артефакт post-run review — чтобы после прогона можно было сравнить прогноз с фактом.
- **T15** Негативная память («так больше не делаем») — зависит от T14; запускается после post-run review. Накопленная память подаётся на вход планировщику (этап Plan), чтобы он не повторял зафиксированных ошибок.
- **T16** LLM substitution — смена LLM на второй попытке rework.

### P3 —  если успеем

- **T17** Визуализатор прогона в TUI — интерактивный просмотр JSONL-лога прогона (зависит от T12).
- **T18** Self-improvement loop — на основе накопленных post-run review и негативной памяти запускается анализ, где оркестратор систематически ошибается; по результатам — прогон по улучшению собственного кода оркестратора (зависит от T14, T15).
- **T19** Кеш установки зависимостей по hash lock-файла.
- **T20** Расширение телеметрии счётчиками токенов — каждый LLM-вызов в JSONL-логе дополняется количеством input/output токенов. Позволяет постфактум посчитать стоимость прогона по прайсу любой модели (зависит от T12).

### Не берём в работу

- Параллельные имплементации.
- Cloud executor.
- Поддержка GitLab / других трекеров — только GitHub и GitHub Issues.
- Разрешение merge-конфликтов.
